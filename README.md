# homebridge-kasa-cam

HomeKit **on/off (privacy) control** for TP-Link **Kasa cameras** (EC70 "Spot Pan Tilt", and other models using the same protocol).

This plugin exposes each camera as a **Switch** in HomeKit:
- **On** = camera enabled
- **Off** = privacy mode (camera disabled, the same as "Camera Off" in the Tapo/Kasa app)

> For the **video stream**, use something like [go2rtc](https://github.com/AlexxIT/go2rtc) (the Kasa local RTSP-ish stream on port 19443) bridged into HomeKit separately. This plugin only handles the **on/off control**, which the stream tools can't do.

## Why it works the way it does

Current Kasa camera firmware (e.g. EC70 `2.3.x`) has **no local control path**:
- UDP/TCP `9999` accepts `get_sysinfo` (read) but **silently ignores** `set_*` commands.
- The HTTPS API on `19443` serves only the video stream â€” no control endpoint (the older KC-series `/data/LINKIE.json` returns `404`).
- The official Tapo app controls the camera **exclusively through the TP-Link cloud** (verified by packet capture: blocking cloud while keeping LAN makes the app's on/off spin forever, and it never attempts a local connection).

So this plugin:
- **Reads** on/off state **locally** over UDP `9999` (`get_sysinfo` â†’ `camera_switch`) when a camera IP is configured â€” fast and offline-friendly.
- **Writes** on/off via the **TP-Link cloud passthrough** API, signed the same way the Tapo app signs its requests (HMAC-SHA1, see below).

## The cloud signing scheme (reverse-engineered)

Every cloud request carries:

```
Content-MD5:    base64(MD5(body))
X-Authorization: Timestamp=9999999999, Nonce=<uuid>, AccessKey=<key>, Signature=<sig>

stringToSign = Content-MD5 + "\n" + "9999999999" + "\n" + Nonce + "\n" + <request-path>
Signature    = hex( HMAC-SHA1(secret, stringToSign) )
```

There are **two key pairs**, bound to the app install's `terminalUUID`:
- **auth** pair â€” signs `/api/v2/account/login` (and other `/account|/common` calls)
- **passthrough** pair â€” signs `/api/v2/common/passthrough`

The device on/off command relayed through passthrough is:

```json
{ "smartlife.cam.ipcamera.switch": { "set_is_enable": { "value": "on" } } }
```

## Configuration

```json
{
  "platforms": [
    {
      "platform": "KasaCam",
      "name": "Kasa Cam",
      "email": "you@example.com",
      "password": "your-tplink-account-password",
      "terminalUUID": "XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX",
      "auth":        { "accessKey": "....", "secret": "...." },
      "passthrough": { "accessKey": "....", "secret": "...." },
      "loginUrl": "https://n-wap.i.tplinkcloud.com",
      "appServerUrl": "https://n-use1-wap.i.tplinkcloud.com",
      "pollSeconds": 30,
      "cameras": [
        { "name": "Living Room", "deviceId": "801A....", "ip": "192.168.3.24", "model": "EC70" },
        { "name": "Bedroom",     "deviceId": "801A....", "ip": "192.168.3.58", "model": "EC70" }
      ]
    }
  ]
}
```

`deviceId` and `ip`: get the deviceId from `get_sysinfo` (or the cloud device list); the IP is your camera's LAN address.

## Capturing `terminalUUID` and the signing key pairs

The `terminalUUID` and the two `accessKey`/`secret` pairs are generated by the Tapo app and are **not** transmitted over the network, so you must capture them once from a device running the Tapo app (the keys persist across sessions for that install):

1. On a **rooted Android** device, run `frida-server`, then launch the Tapo app with a TLS-unpinning script (e.g. `frida -U -f com.tplink.iot -l multiple-unpinning.js`) while proxying through `mitmproxy` (install its CA as a system cert).
2. Hook `javax.crypto.Mac` to capture each `HMAC-SHA1` key + string-to-sign, and read the matching `X-Authorization` from the requests. Match by signature to learn which key signs **login** vs **passthrough**.
3. Read `terminalUUID` from the `termID=` query param / login body.

(See the project history / your capture notes for the exact Frida hooks.)
